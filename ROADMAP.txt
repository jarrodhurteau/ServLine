COMPLETED PHASE: Phase 9 — Structured Variants & Export
================================================================================

Status: COMPLETE (Days 71-85)

  PROBLEM STATEMENT:
  Claude API extracts structured size/variant data (e.g., [{label:"Small",price:10.00},
  {label:"Large",price:20.00}]), but claude_items_to_draft_rows() flattens it into a
  single price_cents integer. The draft_items table has no child structure for variants.
  Variant info is lost between extraction and the editor, making POS export impossible
  without re-parsing ephemeral price_text strings.

  SOLUTION:
  Add a draft_item_variants child table, preserve structured variant data end-to-end
  (extraction -> storage -> editor -> export), and redesign the editor to show
  size/combo sub-rows under each item.

KNOWN ISSUES (discovered during live testing, Mar 4 2026)
..........................................................
  These were identified during a real 200-item pizza menu import after the
  Day 100.5 pipeline cleanup. The Claude extraction path is dramatically
  better than the removed heuristic path, but edge cases remain:

  1. Category merging — "Soups" and "Fresh Buffalo Wings" are separate
     sections on the real menu but Claude merged them into a single
     "Fresh Soups & Buffalo Wings" category. The extraction prompt needs
     better guidance on section boundary detection.

  2. Piece-count variants — Wing items came out as separate items per
     piece count ("12 Pcs Bone-In Wings", "24 Pcs Boneless Wings") instead
     of recognizing piece counts as size variants of "Buffalo Wings" and
     "Boneless Wings". The extraction prompt should detect quantity-based
     pricing tables and emit variants.

  3. Multi-column pricing — The real menu has "plain" and "W/ Fries"
     price columns for wings. These should become variant axes (e.g.,
     kind=style, labels="Plain"/"W/ Fries") but are currently lost or
     partially captured.

  4. Heuristic OCR-only path (ai_preview) — Confirmed completely garbled
     on this menu (names like "OLVLOd", "YSal", "3ma", "VZZid", category
     "Vals"). Validates the Day 100.5 decision to remove it. The old
     segment_document() word-level OCR path produces unusable output.

  Target: Sprint 11.2 prompt engineering + Sprint 11.3 confidence gate
  should catch and correct issues 1-3 via Claude Call 3 reconciliation.

Phase 12 — POS-Native Data Model & Editor (Days 111-127)
---------------------------------------------------------
  CONTEXT: Research into Square, Toast, Clover, Lightspeed, and SpotOn reveals
  that ALL major POS systems structure their menu data the same way:

    Category → Item → Modifier Group → Modifier

  ServLine's current model (Item → Variant with kind tag) is structurally
  flat compared to what POS systems expect. Variants are tagged with a kind
  (size/combo/flavor/style/other) but lack the named container (modifier group)
  with selection rules (required, min/max) that every POS editor uses.

  This mismatch causes export translation problems and prevents the editor
  from matching the mental model restaurant owners develop using their POS.

  GOAL: Align ServLine's data model, editor, and exports with the universal
  POS hierarchy. If the data is already shaped the way POS systems expect,
  exports become trivial format translations rather than structural transforms.

  KEY RESEARCH FINDINGS (from POS competitive analysis):
  ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
  Square: Modifier Sets (list-type or text-type), required/optional, min/max,
    conversational modifiers, per-item overrides. WoFlow integration for
    menu-to-POS import (outsourced). Tile grid POS layout, web forms back office.

  Toast: Modifier Groups (shared across items), nested modifiers (recursive),
    inheritance from menu groups, items double as modifiers. Menu Builder vs
    Classic editor. Change sets with scheduled publishing.

  Clover: Modifier Groups with minRequired/maxAllowed, showByDefault toggle.
    Separate Item Groups concept for variant management (Attributes →
    Attribute Options → Items). Category/tab-based navigation.

  Lightspeed: Production Instructions (free kitchen notes) + Sub-items
    (priced add-ons). Single vs multiple choice at sub-screen level.
    Items cannot be deleted, only archived. Quick Create vs Detailed Create.

  SpotOn: Modifiers organized into sections, any combination can have own price.
    BOH web editor + FOH POS editing. Enterprise multi-location support.

  UNIVERSAL PATTERNS:
  - 4-level hierarchy: Category → Item → Modifier Group → Modifier
  - Modifier groups are always NAMED containers with selection rules
  - Required vs optional + min/max selections is standard everywhere
  - POS layout = color-coded tile grid; back office = web forms/lists
  - Dual naming: customer-facing name, POS/staff name, kitchen name
  - Categories ≠ layout groups (reporting vs display separation)
  - Modifier pricing is always per-modifier (some support group-level)
  - Modifier groups are reusable/shareable across multiple items

KEY PRINCIPLES
================================================================================

1. ONE BRAIN ARCHITECTURE
   - All OCR/AI logic in /storage
   - No duplicate pipelines
   - Single source of truth
   - Core engine has ZERO knowledge of POS systems or delivery targets

2. NO HUMAN REVIEW — EVER
   - No ServLine employees reviewing menus
   - System handles it or asks for better photo
   - Customer (restaurant owner) always does final review
   - Edge cases improve the pipeline, not headcount

3. SAFE AI
   - Never hallucinate prices
   - Never invent categories
   - Preserve original when uncertain
   - 3 Claude calls: extract → verify → reconcile flagged items

4. DETERMINISTIC PROCESSING
   - Same input = same output
   - Full debug artifacts
   - Auditable decisions (changes_made logs from every Claude call)

5. INCREMENTAL PROGRESS
   - Daily commits with verified progress
   - Each phase builds on previous
   - No skipping validation
